## 4.1 Platform Architecture and Compatibility

The **Smart Classroom Assistant** is engineered as a full-stack, cross-platform web application that adheres to a strict client-server architectural separation. The system is designed to be agnostic of the client's operating system, requiring only a modern web browser on the end-user's device. All computationally intensive processes — neural speech transcription, semantic indexing, AI summarization, and assessment generation — are executed exclusively on the server, ensuring a lightweight, accessible, and universally compatible client experience.

### 4.1.1 Browser Compatibility

The application is optimized for Chromium-based browsers, which provide the most robust support for the advanced Web APIs utilized by the system:

- **MediaDevices API**: Enables programmatic access to the host machine's microphone for audio stream capture (teacher interface).
- **WebSocket API (RFC 6455)**: Provides the persistent, full-duplex communication channel supporting real-time transcript delivery and session event synchronization.
- **Web Audio API**: Employed for client-side audio buffer management and stream preprocessing prior to transmission.
- **File API and Fetch API**: Utilized for asynchronous file uploads (teaching materials) and REST endpoint interactions.

**Supported Browsers**: Google Chrome (v110+), Microsoft Edge (v110+), Brave (latest). Firefox offers full functional compatibility; Safari (v16.4+) is supported with minor limitations on audio input device enumeration.

### 4.1.2 Deployment Topology

The system follows a **Three-Tier Architecture**:

```
[Client Tier]          [Application Tier]          [Data Tier]
React SPA         <-->  FastAPI Application  <-->   MongoDB (Primary)
(Browser)              Audio Service               Qdrant (Vector DB)
                       (Whisper Engine)            File Storage
```

- **Client Tier**: The React.js Single Page Application (SPA) runs entirely within the browser. It communicates with the application tier via HTTPS REST calls and secure WebSocket (WSS) connections.
- **Application Tier**: The FastAPI backend orchestrates all business logic. A separate, dedicated Python process — the **Audio Service** — manages the Whisper neural model, operating independently to ensure that audio processing does not contend with API request handling.
- **Data Tier**: MongoDB stores all persistent structured data (user accounts, sessions, assessments). The Qdrant vector database stores semantic text embeddings for the RAG pipeline. The local file system provides material and audio summary storage.

## 4.2 Database Architecture

### 4.2.1 Primary Database: MongoDB

The system uses **MongoDB**, a document-oriented NoSQL database, as its primary persistence layer. The rationale for selecting MongoDB over a traditional relational database system was grounded in the following technical considerations:

- **Schema Flexibility**: Classroom sessions produce heterogeneous data structures — a session document may contain nested arrays of transcript segments, embedded summaries, and pointers to external resources. MongoDB's document model accommodates this variability natively without costly schema migrations.
- **Nested Document Support**: Transcript arrays, concept lists, and assessment question sets are stored as embedded subdocuments, eliminating the need for complex multi-table joins and reducing query latency for common access patterns.
- **High Write Throughput**: Real-time transcript segment persistence demands high-frequency write operations. MongoDB's write-optimized storage engine (WiredTiger) is well-suited to this workload.
- **JSON/BSON Compatibility**: MongoDB's native BSON format aligns seamlessly with the JSON payloads exchanged between the FastAPI backend and the React frontend, eliminating serialization overhead.
- **Horizontal Scalability**: For large institutional deployments, MongoDB's sharding capabilities enable horizontal scaling across multiple database nodes.

## 4.3 Configuration Management

The application employs a secure, layered configuration system to manage all sensitive and environment-specific parameters without embedding them in source code. This separation of configuration from code is a fundamental DevSecOps best practice that ensures the application can be safely deployed across development, staging, and production environments with zero code changes.

All sensitive configuration is stored in a `.env` file in the project root. A centralized `config.py` module loads and validates these values at application startup using `python-dotenv`, making them available as typed Python constants throughout the application.

## 4.4 Neural Transcription Engine (Whisper Small-En)

The academic transcription capability of the Smart Classroom Assistant is powered by a locally deployed implementation of OpenAI's Whisper neural speech recognition architecture. Specifically, the system employs the **`small.en`** model variant — a 244 million parameter English-specialized model — running through the **Faster-Whisper** library, which is a CTranslate2-optimized inference backend that significantly reduces memory footprint and processing latency compared to the original PyTorch implementation.

The deliberate selection of the `small.en` variant reflects a carefully balanced engineering trade-off: the model is sufficiently compact to operate with low latency on server-grade hardware while delivering a Word Error Rate (WER) of approximately 4–6% on standard English academic speech, which is more than adequate for classroom transcription fidelity.

### 4.4.1 Transcription Pipeline Architecture

The transcription pipeline consists of a dedicated `audio_service` process that operates independently from the primary FastAPI application server. This architectural isolation ensures that computationally intensive neural inference does not contend with API request handling.

**Stage 1 — Audio Ingestion:**
The teacher's microphone is accessed via the browser's MediaDevices API. The audio stream is captured in 16-bit PCM format, converted to a **16kHz mono** stream (the input format expected by the Whisper model), and transmitted as binary WebSocket frames to the audio service port.

**Stage 2 — Buffer Accumulation and Segmentation:**
The audio service maintains a rolling audio buffer. Several seconds of audio data are accumulated before inference is triggered, providing the model with sufficient temporal context for accurate speech-to-text conversion. A Voice Activity Detection (VAD) filter is applied to discard silent frames, preventing spurious or empty transcriptions.

**Stage 3 — Neural Inference (Whisper `small.en`):**
The Faster-Whisper implementation processes the audio buffer using the Transformer encoder-decoder architecture:
- The **encoder** ingests the raw audio spectrogram using convolutional feature extraction layers, producing a rich, high-dimensional audio representation.
- The **decoder** performs autoregressive token prediction using **beam search** (beam size 5), iteratively selecting the most probable next token given the current context and the encoder's audio representation.
- The model operates with **FP16 (float16) quantized weights** on GPU, halving memory requirements while maintaining full inference accuracy.

**Stage 4 — Text Segment Post-Processing:**
Generated transcript segments undergo a suite of post-processing operations to eliminate common transcription artifacts:
- **Repetition Suppression**: A sliding-window deduplication algorithm identifies and removes phrases that are repeated verbatim within consecutive segments — a known artifact of streaming transcription.
- **Segment Tracking**: Each segment is assigned a unique processing hash, preventing duplicate segments from being stored in the database or broadcast to clients.
- **Sentence Boundary Normalization**: Segments are normalized to end on grammatically appropriate boundaries where possible.

**Stage 5 — Real-Time Broadcast via WebSocket:**
Cleaned transcript segments are immediately broadcast to all connected students via the session WebSocket channel with sub-500ms end-to-end latency. Simultaneously, each segment is asynchronously persisted to the session document in MongoDB for permanent record-keeping.

### 4.4.2 Performance Characteristics

| Metric | Value |
|:---|:---|
| **Model Parameters** | 244 Million |
| **Model Precision** | FP16 (GPU) / INT8 (CPU fallback) |
| **Input Sample Rate** | 16,000 Hz (16 kHz) Mono |
| **Typical Segment Processing Latency (GPU)** | 200 – 600 ms per segment |
| **Typical Segment Processing Latency (CPU)** | 1,500 – 4,000 ms per segment |
| **Approximate Word Error Rate (Academic English)** | 4 – 8 % |
| **Cold-Start Model Load Time (GPU)** | 3 – 6 seconds |
| **Model Warm Memory Footprint (GPU VRAM)** | ~470 MB |

### 4.4.3 Model Residency and Cold-Start Elimination

To eliminate the "cold-start" latency that would arise from loading the Whisper model from disk on each transcription request, the audio service loads the model weights into GPU memory once at service startup and maintains the model in a resident, warm state throughout the duration of the teaching session. This architectural decision ensures consistent, predictable transcription latency from the first syllable spoken.

## 4.5 Retrieval-Augmented Generation (RAG) Pipeline

The RAG subsystem represents the most architecturally sophisticated component of the Smart Classroom Assistant. It integrates a semantic vector store with a large language model to produce assessments and enable AI-assisted Q&A that are provably grounded in actual lecture content.

### 4.5.1 RAG Architecture Overview

The RAG pipeline consists of three core components:

- **Indexer (`indexer.py`)**: Responsible for extracting, chunking, embedding, and storing text content from lecture transcripts and uploaded PDF materials into the Qdrant vector store.
- **Vector Store (`vector_store.py`)**: A singleton-patterned interface to the Qdrant embedded database, managing collection lifecycle, payload indexing, text embedding via `all-MiniLM-L6-v2`, and semantic search operations.
- **Assessment Engine (`assessment_engine.py`)**: Orchestrates RAG queries and LLM calls to generate assessments and grade student responses.

### 4.5.2 Indexing Pipeline

**Text Chunking**: Source text (from transcripts or PDFs) is segmented into overlapping chunks of **500 characters** with a **100-character overlap** (`chunk_size=500, overlap=100`). The overlap ensures semantic continuity across chunk boundaries, preventing important concepts that span chunk boundaries from being lost during retrieval.

**Embedding Generation**: Each chunk is processed by the `all-MiniLM-L6-v2` Sentence Transformer model to produce a **384-dimensional dense vector embedding**. These embeddings capture the semantic meaning of the text in a high-dimensional space such that semantically similar passages produce vectors with high cosine similarity.

**Storage and Indexing**: Embeddings and their associated metadata (`session_id`, `type`, `chunk_index`, etc.) are stored as `PointStruct` objects in the Qdrant `classroom_knowledge` collection. A **payload index** on the `session_id` field enables high-performance filtered retrieval, ensuring that queries during assessment generation retrieve only chunks belonging to the target session(s).

**Idempotency**: To prevent duplication upon session re-indexing, the system first deletes all existing points matching the target `session_id` before inserting fresh embeddings, ensuring data freshness and consistency.

### 4.5.3 Assessment Generation and Grading via RAG

When a teacher triggers assessment generation for a subject:

1. A broad semantic query (`"key concepts, core definitions, important theories, fundamental principles"`) is executed against the Qdrant vector store, filtered to the relevant session IDs.
2. The top 10–15 most semantically relevant chunks are retrieved.
3. The retrieved chunks are concatenated into a structured prompt and submitted to the **Llama 3.3 70B Versatile** LLM (via Groq's inference API) with explicit instructions to generate a JSON-formatted quiz containing both MCQ and short-answer questions.
4. The generated questions include a `source_evidence` field — a direct excerpt from the lecture context — ensuring every question is traceable to actual instructional content.

For student answer grading:
1. The question text is used as a query to retrieve the 3 most relevant contextual chunks.
2. The question, student answer, and grounding context are submitted to the LLM.
3. The LLM returns a structured JSON response containing a numerical score (0–10), letter grade (A–F), personalized feedback, and a `teacher_quote` — a verbatim excerpt from the lecture that validates or clarifies the student's response.
